# Contributing to Fitness Knowledge Skill

Thank you for your interest in contributing to the Fitness Knowledge Skill! This document provides guidelines and information for contributors.

## 🎯 How to Contribute

### 1. Reporting Issues

If you find a bug or have a suggestion:

1. Check if the issue already exists in GitHub Issues
2. If not, create a new issue with:
   - Clear title
   - Description of the problem or suggestion
   - Steps to reproduce (if applicable)
   - Expected vs actual behavior

### 2. Submitting Changes

1. **Fork** the repository
2. **Create** a feature branch: `git checkout -b feature/amazing-feature`
3. **Make** your changes
4. **Test** your changes (if applicable)
5. **Commit** with clear messages
6. **Push** to your branch: `git push origin feature/amazing-feature`
7. **Submit** a Pull Request

### 3. Pull Request Guidelines

#### Before Submitting

- [ ] Code follows the project's style guidelines
- [ ] Changes are tested (if applicable)
- [ ] Documentation is updated
- [ ] No new warnings or errors

#### PR Description Template

```markdown
## Description
[Describe your changes]

## Type of Change
- [ ] Bug fix
- [ ] New feature
- [ ] Documentation update
- [ ] Other (please describe)

## Testing
[Describe how you tested your changes]

## Checklist
- [ ] My code follows the project's style guidelines
- [ ] I have commented my code, particularly in hard-to-understand areas
- [ ] I have made corresponding changes to the documentation
- [ ] My changes generate no new warnings
- [ ] I have added tests that prove my fix is effective or that my feature works
```

## 📝 Content Guidelines

### Writing Style

- **Clear and concise**: Use simple language
- **Evidence-based**: Cite scientific research when possible
- **Safety-first**: Always include safety considerations
- **Practical**: Provide actionable advice

### Formatting

- Use Markdown for all documentation
- Include headers for organization
- Use bullet points for lists
- Include code examples where helpful

### Content Structure

Each module should include:

1. **Overview**: What the topic covers
2. **Key Concepts**: Important principles
3. **Practical Application**: How to apply the knowledge
4. **Safety Considerations**: Important warnings
5. **References**: Scientific sources (when applicable)

## 🔍 Review Process

### What We Look For

1. **Accuracy**: Is the information correct?
2. **Safety**: Are there any safety concerns?
3. **Clarity**: Is it easy to understand?
4. **Completeness**: Does it cover the topic thoroughly?
5. **Integration**: Does it work with existing content?

### Review Timeline

- Initial review: Within 1 week
- Feedback: Within 2-3 days after review
- Merge: After all feedback addressed

## 🎨 Style Guide

### Writing Style

- Use active voice
- Keep sentences short
- Use bullet points for lists
- Include examples where helpful

### Technical Writing

- Define technical terms
- Use consistent terminology
- Include visual aids when possible
- Provide step-by-step instructions

### Safety Language

- Use "should" for recommendations
- Use "must" for requirements
- Use "avoid" for warnings
- Include disclaimers where appropriate

## 🧪 Testing Changes

### Manual Testing

1. Read through the entire module
2. Check for accuracy
3. Verify safety information
4. Test any exercises described

### Content Testing

1. Follow the instructions exactly
2. Note any confusion or gaps
3. Check for completeness
4. Verify references
